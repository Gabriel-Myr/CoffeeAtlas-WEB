# Miniprogram Bean Discovery Design

**Goal:** 把小程序从“咖啡豆浏览工具”调整为“选豆工具”，让首次进入的用户也能通过清晰的引导路径筛出一支适合自己的豆子。

## 现状

- 小程序首页 `/Users/gabi/CoffeeAtlas-Web/apps/miniprogram/src/pages/index/index.tsx` 当前以产地地图浏览为主，更像内容展示页，不像明确的选豆入口。
- 发现主流程已经存在于 `/Users/gabi/CoffeeAtlas-Web/apps/miniprogram/src/pages/all-beans/index.tsx`，并且已经接入：
  - `/api/v1/beans/discover`
  - `/api/v1/beans`
  - 上新筛选能力
  - 收藏驱动的个性化筛选
- 当前发现顺序是“处理法 -> 大洲 -> 国家 -> 结果”，与产品讨论里确定的“产地优先”方向不一致。
- 本地存储 `/Users/gabi/CoffeeAtlas-Web/apps/miniprogram/src/utils/storage.ts` 已管理登录态、收藏、历史，但没有用户分层信息，无法区分新手与熟手。

## 采用方案

### 方案 B（采用）

本期只做小程序，聚焦 4 个变化：

1. 首页改成“选豆入口页”
2. 增加“新手 / 熟手”分流
3. 发现流程改成“产地优先”
4. 保留新品与个性化筛选，但优化表达和入口关系

这样可以最大限度复用现有 discover、新品、收藏、推荐能力，不需要新后端接口，也不需要先做社区功能。

### 不采用的方案

- 继续保留地图首页，只在发现页里微调顺序
  - 用户第一次进入时仍然感知不到产品主线
- 先做社区、评分、记录
  - 成本高，且无法解决“为什么要用这个小程序来选豆”这一核心问题
- 新增独立 onboarding 页面
  - 结构更完整，但对当前阶段偏重，不如先用轻量本地分流验证方向

## 产品定位

一句话定义：

帮助用户从复杂的电商商品信息里，更快找到适合自己的咖啡豆，并在筛选过程中顺便理解产地与处理法。

## 目标用户

### 1. 新手

- 不知道怎么选豆
- 对风味、处理法、产地缺乏概念
- 更需要清晰路径和短解释

### 2. 熟手

- 已经对咖啡豆有一定认知
- 想快速用产地、国家、处理法来缩小结果范围

### 3. 回访用户

- 更关心上新
- 期待基于收藏得到更贴近偏好的新品筛选入口

## 本期范围

### 要做

- 重构首页信息架构，让“带我选豆 / 最近上新 / 直接筛豆”成为主入口
- 新增本地 onboarding 状态，记录用户是新手还是熟手
- 将 discover 交互顺序调整为“大洲 -> 国家 -> 处理法 -> 结果”
- 为 discover 各步骤补充一句简短解释文案
- 优化新品筛选栏文案，让“基于收藏 / 热门新品”表达更明确

### 不做

- 社区评论与评分
- 个人打卡记录体系
- 烘焙商关注流
- 新后端接口或数据库结构调整

## 核心路径

### 路径 A：新手

1. 进入首页
2. 点击“带我选豆”
3. 首次被问“你更像哪类用户”
4. 选择“我是新手”
5. 进入 discover，按“大洲 -> 国家 -> 处理法 -> 结果”走
6. 查看推荐豆和完整结果

### 路径 B：熟手

1. 进入首页
2. 点击“直接筛豆”
3. 进入 discover
4. 快速筛选并查看结果

### 路径 C：回访用户

1. 进入首页
2. 点击“最近上新”
3. 打开新品列表
4. 用基于收藏生成的筛选项继续找豆

## 页面设计

### 1. 首页 `/pages/index`

首页从地图浏览页改成选豆入口页。

建议结构：

- 顶部定位文案
  - 例如：“不知道怎么买豆？从产地开始选。”
- 3 个入口卡片
  - `带我选豆`
  - `最近上新`
  - `我有点了解，直接筛`
- 次级内容模块
  - 保留现有 Atlas 地图浏览能力
  - 作为“按产地逛一逛”的内容区，而非主 CTA

首页的任务不是承载全部信息，而是让用户一眼知道应该从哪里开始。

### 2. 发现页 `/pages/all-beans`

discover 是本期真正的核心页。

现有结构可复用，但顺序要调整为：

1. 第一步：大洲
   - 文案：“先确定一个风土大方向。”
2. 第二步：国家
   - 文案：“再缩小到更具体的产区特征。”
3. 第三步：处理法
   - 文案：“最后决定你更想要哪种口感表现。”
4. 第四步：结果
   - 保留当前结果列表、推荐豆、空状态、分页加载

这样可以让产品表达与“产地优先”的定位一致。

### 3. 我的 `/pages/profile`

本期只需要给用户一个入口，用来：

- 查看当前 onboarding 身份
- 必要时重置“新手 / 熟手”选择

不用扩展成复杂设置页。

### 4. 新品区域

逻辑保持现状，重点优化文案表达：

- `personalized` -> “基于你的收藏”
- `mixed` -> “收藏方向 + 热门新品”
- `fallback` -> “热门新品”

目标是让用户更容易理解为什么会看到这些筛选项。

## 状态设计

新增本地 onboarding 状态，保存在 `storage.ts`。

建议字段：

- `experienceLevel: 'beginner' | 'intermediate'`
- `completedAt: number`

用途：

- 首页决定展示文案与入口行为
- discover 决定默认提示文案和首屏引导
- profile 提供重置入口

## 跳转与参数

为了避免新建复杂全局状态，本期优先用“本地状态 + 页面跳转参数”组合：

- 首页进入 discover 时，用 query 参数表达入口意图
  - 例如 `entry=guided`
  - `entry=direct`
  - `tab=new`
- discover 页根据参数决定：
  - 默认激活 `discover` 还是 `new`
  - 是否在首次进入时显示 onboarding 提示

## 数据与接口

本期不新增接口。

继续复用：

- `/api/v1/beans/discover`
- `/api/v1/beans`
- `/api/v1/beans/new-arrivals/filters`

原因：

- 当前后端 discover 已支持 `process`、`continent`、`country` 组合筛选
- 当前问题主要是前端流程表达，不是接口能力不足

## 测试策略

本期重点做前端状态与辅助逻辑测试，不要求为页面渲染新增重型测试基础设施。

建议覆盖：

- `storage.ts` 新增 onboarding 状态读写
- `new-arrival-filters.ts` 现有逻辑不回归
- 新增 discover 步骤顺序或入口参数解析的纯函数测试

同时至少跑：

- `pnpm --filter @coffeeatlas/miniprogram test`
- `pnpm --filter @coffeeatlas/miniprogram typecheck`

## 风险与控制

### 风险 1：首页和 discover 重构范围过大

控制：

- 不新建独立 onboarding 页面
- 不改后端接口
- 先用本地状态和参数驱动流程

### 风险 2：现有 discover 代码已经较大，继续堆逻辑会更难维护

控制：

- 本期允许在计划中把 discover 页拆出少量 helper
- 但不做与目标无关的大重构

### 风险 3：用户分流做得过重，反而打断首屏体验

控制：

- 首次仅在用户点“带我选豆”时弹出轻量分流
- 不是一打开小程序就强制拦截

### 风险 4：新品能力被边缘化

控制：

- 首页保留“最近上新”主入口
- 新品 tab 保持为一级可达能力

## 默认假设

- 本期只做小程序，不改 Web
- 当前工作重点是把“浏览”改成“选豆”
- 产地优先是本期明确方向，因此 discover 需要围绕“大洲 -> 国家 -> 处理法”重排
- 价格筛选暂不纳入本期交互主线，后续可单独规划二期
